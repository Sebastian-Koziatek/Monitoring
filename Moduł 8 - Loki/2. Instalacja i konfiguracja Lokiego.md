# Instalacja i konfiguracja Loki

Krótki, praktyczny przewodnik instalacji **Loki 2.9.3** na **Ubuntu 24.04** z binarki jako usługa systemd. Na końcu: zbieranie logów przez Promtail i rozwiązywanie problemów.

> **Loki w jednym zdaniu:** to „Prometheus dla logów" — magazyn logów, który indeksuje **tylko etykiety** (nie treść), a odpytujesz go w Grafanie językiem LogQL. Sam nie czyta plików — logi wysyła do niego agent (Promtail).

## Wymagania

- Ubuntu 24.04, uprawnienia `sudo`, dostęp do internetu
- Min. 512 MB RAM (zalecane 2 GB), ~10 GB dysku
- Narzędzia: `sudo apt install -y wget unzip curl`

---

## Instalacja z binarki (systemd)

### 1. Pobierz binarkę

```bash
cd /tmp
wget https://github.com/grafana/loki/releases/download/v2.9.3/loki-linux-amd64.zip
unzip loki-linux-amd64.zip
sudo mv loki-linux-amd64 /usr/local/bin/loki
sudo chmod +x /usr/local/bin/loki
loki --version   # sprawdzenie
```

### 2. Użytkownik systemowy i katalogi

```bash
sudo useradd --system --no-create-home --shell /bin/false loki
sudo mkdir -p /var/lib/loki/{index,cache,chunks,wal,compactor} /etc/loki
sudo chown -R loki:loki /var/lib/loki /etc/loki
```

Loki działa jako dedykowany użytkownik `loki`, a wszystkie dane trzyma w `/var/lib/loki` (m.in. `wal/` — Write-Ahead Log, dzięki któremu logi nie giną przy restarcie/awarii).

### 3. Konfiguracja — `/etc/loki/loki-config.yaml`

```yaml
auth_enabled: false                # tryb single-tenant (bez nagłówka X-Scope-OrgID)

server:
  http_listen_port: 3100           # API: push logów, zapytania, /ready, /metrics
  grpc_listen_port: 9096

ingester:
  lifecycler:
    address: 127.0.0.1
    ring:
      kvstore: { store: inmemory }
      replication_factor: 1
  chunk_idle_period: 5m
  max_transfer_retries: 0          # MUSI być 0 przy włączonym WAL
  wal:
    enabled: true                  # ochrona przed utratą danych
    dir: /var/lib/loki/wal

schema_config:
  configs:
    - from: 2020-10-24
      store: boltdb-shipper
      object_store: filesystem
      schema: v11
      index: { prefix: index_, period: 24h }

storage_config:
  boltdb_shipper:
    active_index_directory: /var/lib/loki/index
    cache_location: /var/lib/loki/cache
    shared_store: filesystem
  filesystem:
    directory: /var/lib/loki/chunks

limits_config:
  reject_old_samples: true
  reject_old_samples_max_age: 168h  # odrzucaj logi starsze niż 7 dni

compactor:
  working_directory: /var/lib/loki/compactor
  shared_store: filesystem
```

> **Uwaga o wersji:** konfiguracja jest poprawna dla **2.9.3**. Pola `shared_store` zostały **usunięte w Loki ≥ 2.10** — jeśli kiedyś zaktualizujesz, usuń oba wystąpienia `shared_store`. Retention (auto-usuwanie logów) jest domyślnie wyłączony — do produkcji dodasz sekcję `table_manager`.

### 4. Usługa systemd — `/etc/systemd/system/loki.service`

```ini
[Unit]
Description=Loki Log Aggregation System
After=network.target

[Service]
User=loki
Group=loki
WorkingDirectory=/var/lib/loki
ExecStart=/usr/local/bin/loki -config.file=/etc/loki/loki-config.yaml
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

`WorkingDirectory=/var/lib/loki` jest **konieczne** — bez tego ingester pada na `shutdown-requested.txt: permission denied` i usługa nie wstaje.

### 5. Start i weryfikacja

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now loki
sudo systemctl status loki
curl http://localhost:3100/ready       # po ~15-20 s zwróci: ready
```

---

## ⚠️ Trzy pułapki, które ratują przed „Loki nie działa"

1. **Uruchamiaj przez `systemctl`, nigdy ręcznie ze swojego konta** — Loki musi działać jako użytkownik `loki`. Ręczny start daje `mkdir /var/lib/loki/...: permission denied`.
2. **`WorkingDirectory=/var/lib/loki` w unicie** — inaczej `shutdown-requested.txt: permission denied`.
3. **`/ready` przez pierwsze ~15 s zwraca `Ingester not ready: waiting for 15s`** — to normalne, poczekaj i powtórz.

## Zbieranie logów — Promtail

Loki nie czyta plików samodzielnie. Promtail monitoruje pliki logów, dodaje etykiety i wysyła je do Loki.

### Instalacja

```bash
cd /tmp
wget https://github.com/grafana/loki/releases/download/v2.9.3/promtail-linux-amd64.zip
unzip promtail-linux-amd64.zip
sudo mv promtail-linux-amd64 /usr/local/bin/promtail
sudo chmod +x /usr/local/bin/promtail
sudo mkdir -p /etc/promtail /var/lib/promtail
```

### Konfiguracja — `/etc/promtail/promtail-config.yaml` (przykład: logi auditd)

```yaml
server:
  http_listen_port: 9080
  grpc_listen_port: 0

positions:
  filename: /var/lib/promtail/positions.yaml   # zapamiętuje, dokąd już wysłano

clients:
  - url: http://localhost:3100/loki/api/v1/push

scrape_configs:
  - job_name: audit
    static_configs:
      - targets: [localhost]
        labels:
          job: audit
          __path__: /var/log/audit/audit.log
    pipeline_stages:
      - regex: { expression: '^type=(?P<audit_type>\w+)' }    # etykieta na KAŻDEJ linii
      - regex: { expression: 'key="(?P<audit_key>[^"]+)"' }   # audit_key tam, gdzie występuje
      - labels:
          audit_type:
          audit_key:
```

> **Dlaczego dwa osobne `regex`?** Pole `key="..."` jest tylko w części linii auditd (ok. 20%). Gdyby jeden regex wymagał `key=`, to ~80% linii nie dostałoby nawet etykiety `audit_type` i zapytania typu `{audit_type="EXECVE"}` zwracałyby pustkę. Rozbicie na dwa stage'e daje `audit_type` na **każdej** linii.

### Usługa systemd — `/etc/systemd/system/promtail.service`

```ini
[Unit]
Description=Promtail Log Agent
After=network.target auditd.service

[Service]
User=root                          # potrzebny dostęp do /var/log/audit/audit.log
ExecStart=/usr/local/bin/promtail -config.file=/etc/promtail/promtail-config.yaml
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now promtail
curl http://localhost:9080/ready
```

## Szybki test (bez Promtail)

```bash
# wyślij testowy log
curl -s -XPOST http://localhost:3100/loki/api/v1/push \
  -H "Content-Type: application/json" \
  --data-raw '{"streams":[{"stream":{"job":"test"},"values":[["'$(date +%s)000000000'","hello loki"]]}]}'

# odpytaj o niego
curl -sG http://localhost:3100/loki/api/v1/query --data-urlencode 'query={job="test"}'
```

Odpowiedź z logiem `hello loki` oznacza, że Loki działa poprawnie.

## Rozwiązywanie problemów

Diagnostyka: `sudo journalctl -u loki -n 50 --no-pager` oraz `loki -config.file=/etc/loki/loki-config.yaml -verify-config`.

| Błąd w logach | Przyczyna | Rozwiązanie |
|---|---|---|
| `mkdir /var/lib/loki/...: permission denied` | Loki działa jako zły użytkownik / brak `chown` | Startuj przez systemd (`User=loki`), wykonaj `chown` z kroku 2 |
| `shutdown-requested.txt: permission denied` | Brak `WorkingDirectory` w unicie | Dodaj `WorkingDirectory=/var/lib/loki` |
| `max_transfer_retries has to be 0 when WAL is enabled` | Usunięto `max_transfer_retries: 0` | Przywróć `max_transfer_retries: 0` |
| `field shared_store not found` | Konfiguracja 2.9.x na Loki ≥ 2.10 | Usuń oba `shared_store` |
| `bind: address already in use` | Port 3100 zajęty | `sudo ss -tlnp \| grep 3100`, zatrzymaj kolidujący proces |

**Złota zasada:** nie uruchamiaj `loki` ręcznie ze swojego konta — zawsze `sudo systemctl start loki`.

## Ściągawka

| Element | Lokalizacja |
|---|---|
| Binarka | `/usr/local/bin/loki` |
| Konfiguracja | `/etc/loki/loki-config.yaml` |
| Dane | `/var/lib/loki/` |
| Usługa | `/etc/systemd/system/loki.service` |

```bash
sudo systemctl {start|stop|restart|status} loki
sudo journalctl -u loki -f
curl http://localhost:3100/ready       # gotowość
curl http://localhost:3100/metrics     # metryki
```

**Porty:** `3100` (HTTP API), `9096` (gRPC).

**Kolejny krok:** dodanie Loki jako źródła danych w Grafanie i budowa dashboardów opartych o logi (kolejne lekcje modułu).
